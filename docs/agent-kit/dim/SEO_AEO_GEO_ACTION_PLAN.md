# DIM — SEO / AEO / GEO ACTION PLAN

## 목적

이 문서는 `SEO / AEO / GEO`를 DIM 현재 단계에 맞는 실행 언어로 바꾼 문서다.

핵심 전제는 단순하다.

- SEO, AEO, GEO는 별도 채널이 아니다
- 하나의 visibility system 이 서로 다른 표면에서 보이는 방식이다
- 지금 DIM에 필요한 것은 페이지 수 확대가 아니라 `지식 객체 품질` 강화다

## DIM 현재 상태

DIM은 현재 아래 상태까지 와 있다.

- 공개 라우트 구성 완료
- MDX 기반 피처 운영
- 대표 이미지 체계 반영
- `Organization` / `Article` JSON-LD 기본 구현
- preview / production 배포 경로 분리
- 홈과 `/articles`의 매거진형 아카이브 리디자인 완료
- `/about`, `/submit`, `/articles/[slug]` 1차 통일 완료

즉, 지금 DIM은 `검색에 잡히기 위한 최소 기술 조건`을 다시 만드는 단계가 아니라,
`기존 피처를 더 answerable / citable / trustworthy 하게 바꾸는 단계`다.

## DIM용 5단 프레임

### 1. Eligibility

엔진이 DIM 페이지에 접근하고 인덱스할 수 있는가

현재 DIM 체크 항목:

- `200` 응답
- indexable HTML
- robots / sitemap / canonical
- preview와 production 경로 정합성
- 이미지와 메타 경로 정합성

### 2. Relevance

제목, H1, 본문, 섹션 구조가 같은 질문과 의도를 가리키는가

DIM 번역:

- 피처 제목이 단순히 멋있는 문장이 아니라 검색 가능한 핵심 문장인가
- 홈과 `/articles` 인트로가 각 피처 묶음의 의미를 설명하는가
- 내부 링크와 앵커가 실제 주제를 설명하는가

### 3. Extractability

문단이 답변 단위로 잘 잘리는가

DIM 번역:

- 피처 도입부가 `무엇인지 / 왜 중요한지 / 어떤 구조 변화인지`를 바로 말하는가
- 소제목 아래 1~2문장 direct answer가 있는가
- 비교, 정의, 절차, 예외가 블록 단위로 분리돼 있는가

### 4. Groundability

주장이 근거와 엔터티 정보로 뒷받침되는가

DIM 번역:

- 브랜드명, 서비스명, 공식 링크가 초반에 분명한가
- 판단문 옆에 근거 문장, 비교 지점, 수치, 출처가 붙어 있는가
- DIM과 작성 주체가 기계에도 분명히 보이는가

### 5. Convertibility

노출 뒤에 다음 행동으로 이어지는가

DIM 번역:

- 피처를 읽은 뒤 관련 피처로 더 들어갈 수 있는가
- 피처 제안으로 자연스럽게 이동할 수 있는가
- DIM이 어떤 매거진인지 바로 이해되는가

## 지금 당장 해야 할 것

### 1. 기사 템플릿을 answer-ready 하게 정리

모든 피처에 아래 슬롯을 고정한다.

1. 제목
2. 짧은 해석
3. 첫 문단: `누가 / 무엇을 / 왜 지금`
4. 근거 문장 또는 비교 문장
5. 관련 피처 이동

### 2. 도입부 문장 규칙 통일

각 피처 첫 1~2문단 안에 반드시 아래를 넣는다.

- 공식 브랜드명/서비스명
- 그것이 무엇인지 한 문장 설명
- 왜 지금 중요한지 한 문장
- 가능하면 공식 링크

### 3. evidence density 강화

DIM 피처는 형용사보다 아래를 더 많이 가져야 한다.

- 정의
- 비교
- 수치
- 기준 시점
- 공식 링크
- 출처 문장
- 재인용 가능한 판단문

### 4. 내부 링크를 허브처럼 설계

- 홈 -> `/articles` -> 상세 -> 관련 피처 -> 다시 허브
- 관련 피처는 랜덤이 아니라 같은 판단 축으로 묶기
- 앵커는 설명형으로 쓰기

### 5. 메타와 visible text 정합성 점검

- title / description / OG / canonical
- JSON-LD와 실제 화면 문구
- 대표 이미지 URL
- 카테고리명과 articleSection

## 런칭 직전 해야 할 것

### 1. Search Console / Bing Webmaster 연결

- `depthintelligence.kr` 등록
- sitemap 제출
- 대표 피처 URL 색인 요청

### 2. 브랜드 검색 테스트

아래 유형으로 직접 확인한다.

- `[브랜드명]`
- `[브랜드명 서비스]`
- `[브랜드명 제품]`
- `[브랜드명 런칭]`
- `[브랜드명 분석]`

### 3. 피처 메타 최종 고정

- article title
- excerpt
- OG image
- canonical
- schema visible-text 정합성

### 4. preview / production 정합성 검수

- 경로 구조
- 이미지 경로
- 메타 경로
- robots / sitemap

### 5. 고가치 페이지 우선 점검

DIM에서 런칭 직전 우선순위는 아래다.

- 홈
- `/articles`
- 대표 피처 2~3개
- `/about`
- `/submit`

## 런칭 후 운영 리듬

### 월간

- top query 검토
- high-impression / low-engagement 피처 재작성
- 오래된 링크/출처 점검
- 메타/구조화 데이터 검수

### 분기

- 반복 등장 주제 허브 설계
- 브랜드/서비스 entity page 후보 정리
- comparison / glossary / companion page 후보 정리

### 누적 기준

- 피처 수보다 `answer-ready feature` 비율
- 클릭 수보다 `질 높은 방문`과 재방문
- 브랜드 검색에서 DIM 피처가 같이 잡히는지 여부

## 페이지별 적용 포인트

### Home

- 인트로가 DIM의 질문과 판단 축을 더 분명히 말해야 한다
- 단순 소개보다 `이 매거진이 어떤 질문에 답하는가`가 먼저 보여야 한다

### `/articles`

- 아카이브 소개 문구를 topic cluster 입구처럼 정리
- 카테고리 채널 바를 향후 허브 구조의 출발점으로 사용

### Article Detail

- 제목 아래 `짧은 해석` 유지
- 본문 초반에 direct answer 성격의 핵심 문장 추가
- 비교/정의/예외/근거 블록을 더 분명하게 분리

### `/about`

- DIM의 entity trust layer 역할을 강화
- 편집 기준, 책임 주체, 방식이 기계와 사람 모두에게 분명해야 한다

### `/submit`

- 제안은 그대로 실리지 않는다는 원칙 유지
- 어떤 정보를 받는지보다, 어떤 정보가 citation-ready feature로 바뀔 수 있는지가 보여야 한다

## 피처 템플릿 규칙

1. 첫 문단 안에 브랜드명과 서비스 설명을 넣을 것
2. 첫 화면 안에 `왜 중요한가`를 직접 말할 것
3. 한 문단에는 한 주장만 둘 것
4. 비교와 예외는 표/리스트/짧은 블록으로 나눌 것
5. 근거 문장 없는 판단문은 남발하지 말 것
6. PDF/이미지에만 핵심 정보를 넣지 말 것
7. 공식 링크와 책임 주체를 숨기지 말 것

## 하지 말아야 할 것

- keyword stuffing

- 근거 없는 해석 과장

## 한 줄 결론

DIM의 다음 단계는 더 많은 페이지를 만드는 것이 아니라,
지금 있는 피처를 `검색 가능 + 답변 가능 + 인용 가능 + 신뢰 가능`한 지식 객체로 바꾸는 일이다.
