## 2026-03-13
### 현재 단계
초기 재구축 방향 확정

### 결정한 내용
- 스택은 `Next.js 16 + TypeScript + Cloudflare Workers`
- review preview는 Cloudflare Pages `dim-preview`
- 콘텐츠는 1차에서 로컬 MDX 기반으로 운영
- D1, R2, CMS, 관리자 기능은 보류

### 왜 이렇게 결정했는가
- 공개 버전은 운영 시스템보다 브랜드와 읽기 경험이 더 중요했기 때문

### 기대효과
- 구현 복잡도를 낮추고 공개 품질에 집중

### 보류한 것
- CMS
- D1 / R2 연결
- 관리자 기능

### 다음 액션
- 공개 라우트와 샘플 피처 완성

## 2026-03-14
### 현재 단계
공개 제품 프레이밍 정리

### 결정한 내용
- DIM은 뉴스룸이 아니다
- DIM은 제보 플랫폼이 아니다
- 공개 프레이밍은 `프로젝트 제출 -> 편집 및 해석 -> 콘텐츠 발행`
- 제출 페이지는 `피처 제안` 관점으로 유지

### 왜 이렇게 결정했는가
- 실제 제품 논리가 브랜드/서비스/프로젝트 정보를 받아 편집 피처로 재구성하는 구조이기 때문

### 기대효과
- 뉴스룸/운영문서/제보 서비스 같은 인상을 줄임

### 보류한 것
- 실제 접수 처리 방식

### 다음 액션
- 제출 페이지와 공개 카피 정리

## 2026-03-15
### 현재 단계
매거진 문법 전면 재구성

### 결정한 내용
- 9works 매거진 페이지의 구조 문법을 reference로 사용
- Home과 `/articles`를 dark intro + category bar + archive grid 구조로 재구성
- listing card는 이미지 + 제목 + 짧은 요약 중심으로 단순화
- `/about`, `/submit`도 같은 매거진 세계 안으로 재정리

### 왜 이렇게 결정했는가
- 기존 DIM 구조가 섹션 누적형 랜딩처럼 보여, 매거진 아카이브 인상이 약했기 때문

### 기대효과
- 전체가 하나의 매거진 시스템처럼 보이도록 정리

### 보류한 것
- DIM 고유 카피와 카테고리 감도 최종 조정

### 다음 액션
- reference grammar 위에 DIM 전용 정체성 좁히기

## 누적 핵심 판단

### 편집/철학
- DIM은 정보 전달보다 해석 권위를 먼저 본다
- 제출은 그대로 게시하지 않고 피처로 다시 정리한다

### 권위/아카이브
- 검색은 결과이고 권위 축적이 원인이다
- 목록 구조와 아카이브 연결이 중요하다

### 구현/운영
- 지금은 in-place refinement가 기본 전략이다
- preview와 production은 반드시 분리 운영한다

### 수익/확장
- 조기 광고/스폰서십보다 권위 형성이 우선이다
