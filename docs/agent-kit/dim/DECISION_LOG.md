# DIM — DECISION LOG

## 기록 원칙

이 문서는 DIM의 편집, 제품, 운영 판단을 남기는 로그다.
모든 기록은 `권위 축적`, `아카이브 가치`, `현재 구현 가능성` 기준으로 남긴다.

## 2026-03-08
### 현재 단계
초기 공개 버전 방향 확정 단계

### 결정한 내용
- DIM은 뉴스룸이 아니라 에디토리얼 인텔리전스 매거진으로 정의한다
- `프로젝트 제출 -> 편집 및 해석 -> 콘텐츠 발행` 흐름을 기본 제품 논리로 고정한다

### 왜 이렇게 결정했는가
- 제보/뉴스룸 프레이밍은 DIM의 권위와 어긋난다
- 제출 기반 편집 구조를 공개 제품 안에서 자연스럽게 보여줄 필요가 있었다

### 기대효과
- 브랜드 정체성과 제품 구조가 충돌하지 않는다
- 제출 페이지와 공개 피처가 하나의 시스템으로 읽힌다

### 보류한 것
- 실제 제출 접수 기능 연결

### 다음 액션
- public Korean wording 정리

## 2026-03-13
### 현재 단계
v1 공개 범위 확정 단계

### 결정한 내용
- v1 공개 라우트를 `/`, `/articles`, `/articles/[slug]`, `/about`, `/submit`로 고정한다
- 초기 운영은 MDX 기반으로 가고 CMS/D1/R2는 phase 2로 미룬다
- 리뷰용 preview는 Pages, 실제 운영 배포는 Workers로 분리한다

### 왜 이렇게 결정했는가
- 지금은 운영 도구보다 공개 브랜드 경험이 더 중요했다
- 배포 경로를 분리해야 실제 도메인과 리뷰 링크가 충돌하지 않는다

### 기대효과
- 제품 범위가 과도하게 무거워지지 않는다
- 리뷰와 운영 환경을 분리해 실수 가능성을 줄인다

### 보류한 것
- 관리자/CMS
- 검색
- 회원/결제

### 다음 액션
- 공개 UI 정교화

## 2026-03-15
### 현재 단계
카피/전환/제안 UX 보정 단계

### 결정한 내용
- `시장 신호`는 현재 홈과 `/articles` 채널 바에서 비노출 처리한다
- 제출 페이지는 `피처 제안` 중심으로 단순화한다
- 페이지 전환은 복잡한 오버레이 대신 mirrored fade 방향으로 맞춘다

### 왜 이렇게 결정했는가
- 현재 채널 바는 3개 축이 더 명확했다
- 제출 페이지는 설명이 많을수록 신뢰가 떨어졌다
- 전환은 레퍼런스에 가까운 단순 페이드가 더 맞았다

### 기대효과
- 첫 화면과 아카이브가 더 정돈된다
- 제출 입구가 덜 문서적이고 더 매거진답게 보인다

### 보류한 것
- 전환 감각의 세부 최종 보정

### 다음 액션
- 전환 최종 검수
- 채널/카피 DIM화

## 2026-03-18
### 현재 단계
매거진 문법 리디자인 완료 단계

### 결정한 내용
- 9works 레퍼런스 문법을 번역한 in-place redesign을 채택한다
- 홈과 `/articles`를 dark intro + centered channel bar + uniform archive grid 체계로 통일한다
- `/about`, `/submit`, `/articles/[slug]`도 같은 브랜드 세계 안에서 다시 정리한다

### 왜 이렇게 결정했는가
- 기존 구조는 설명형 블록이 많아 매거진보다 정리된 사이트처럼 읽혔다
- 하나의 매거진 시스템으로 묶어야 DIM의 에디토리얼 인상이 더 선명해진다고 판단했다

### 기대효과
- 홈과 아카이브의 문법이 통일된다
- 이후 DIM 고유 카피와 카테고리 톤을 좁히기 쉬워진다

### 보류한 것
- DIM 고유성의 추가 정교화

### 다음 액션
- 카피, 카테고리, 기사 상세 리듬 보정

## 누적 핵심 판단

### 편집/철학
- DIM은 뉴스룸이 아니다
- DIM은 프로젝트 제출을 받아 편집과 해석을 거쳐 피처로 정리한다
- 해석 권위가 검색 유입보다 먼저다

### 권위/아카이브
- 홈과 `/articles`는 하나의 매거진 시스템으로 움직여야 한다
- 피처는 단발성 소개가 아니라 축적 자산이어야 한다

### 구현/운영
- MDX 기반 운영을 유지하고 무거운 CMS는 미룬다
- Preview는 Pages, production은 Workers로 분리한다

### 수익/확장
- 권위 형성 전 수익화는 보류한다
- 제출 기반 편집 구조는 장기 확장 입구로 본다
