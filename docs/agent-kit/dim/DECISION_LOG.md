## 2026-03-08
### 현재 단계
초기 공개 버전 방향 확정 단계

### 결정한 내용
- DIM은 뉴스룸이 아니라 에디토리얼 인텔리전스 매거진으로 정의한다
- `프로젝트 제출 -> 편집 및 해석 -> 콘텐츠 발행` 흐름을 기본 제품 논리로 고정한다

### 왜 이렇게 결정했는가
- 제보/뉴스룸 프레이밍은 DIM의 권위와 어긋난다
- 제출 기반 편집 구조를 공개 제품 안에서 자연스럽게 보여줄 필요가 있었다

### 기대효과
- 브랜드 정체성과 제품 구조가 충돌하지 않는다
- 제출 페이지와 공개 피처가 하나의 시스템으로 읽힌다

### 보류한 것
- 실제 제출 접수 기능 연결

### 다음 액션
- public Korean wording 정리

- 리뷰와 운영 환경을 분리해 실수 가능성을 줄인다

## 2026-03-19
### 현재 단계
SEO / AEO / GEO 적용 우선순위 정리 단계

### 결정한 내용
- DIM은 SEO, AEO, GEO를 별도 채널이 아니라 하나의 visibility system으로 본다
- 현재 우선순위는 새 페이지 확대보다 `answer-ready / citation-ready feature` 정리다
- direct answer, evidence sentence, entity clarity를 피처 템플릿 핵심으로 둔다

### 왜 이렇게 결정했는가
- 현재 DIM은 기술적 바닥과 기본 메타 구조는 이미 확보했다
- 다음 성장은 키워드 반복이 아니라 답변 단위, 근거 단위, 엔터티 단위 품질에서 나온다고 판단했다

### 기대효과
- 브랜드 검색과 주제 검색에서 DIM 피처가 더 잘 이해되고 인용될 가능성이 높아진다
- 기사 상세 품질과 제출 기반 편집 구조가 하나의 운영 언어로 묶인다

### 보류한 것
- 허브 페이지 실제 구현
- Search Console / Bing Webmaster 운영 루프 실배포 연결

### 다음 액션
- 피처 템플릿 규칙 문서화
- 기사 상세 상단과 도입부 정교화

## 2026-03-19
### 현재 단계
내부 편집 시스템 설계 착수 단계

### 결정한 내용
- DIM 내부 시스템은 일반 CMS가 아니라 `proposal intake -> normalization -> AI draft -> editorial approval -> publication` 파이프라인으로 설계한다
- source of truth는 `D1`, 자산 저장은 `R2`, 비동기 오케스트레이션은 `Queues`를 기본 선택으로 둔다
- `제안 검토`와 `기사 작성`은 같은 화면에 섞지 않고, `/admin/inbox`와 `/admin/drafts`로 분리한다
- 원본 제출과 AI 산출물, 최종 편집본은 서로 다른 데이터 레이어로 보관한다

### 왜 이렇게 결정했는가
- 현재 공개 제품은 이미 발행면보다 intake 시스템이 더 중요한 단계에 들어섰다
- DIM의 권위는 자동 발행이 아니라 검토 가능한 초안과 인간 판단에서 나온다
- 내부 시스템이 제안 원본과 편집 결과를 섞으면 향후 신뢰, SEO/AEO/GEO, 운영 감사성이 동시에 무너진다

### 기대효과
- 제출 데이터와 자료를 안전하게 보관하면서 바로 triage 가능한 editorial inbox를 만들 수 있다
- AI는 초안 보조 역할에 머물고, 최종 publish/no-publish 판단은 편집자에게 남는다
- 이후 entity hub, 반복 브랜드/주제 누적, publication snapshot 구조로 자연스럽게 확장 가능하다

### 보류한 것
- 실제 auth 방식
- 외부 AI provider 확정
- Vectorize 같은 semantic layer 도입

### 다음 액션
- D1 schema 초안 정리
- `/submit` API contract 설계
- `/admin/inbox`, `/admin/proposals/[id]`, `/admin/drafts/[id]` wireframe 수준 명세 작성

## 2026-03-19
### 현재 단계
에이전트 협업 방식 고정 단계

### 결정한 내용
- DIM은 기존 에이전트들과 지속 협업하며, 결과를 다음 라운드의 입력으로 다시 투입하는 재귀적 자기개선 루프를 기본 운영 원칙으로 삼는다
- 새 에이전트는 기존 DIM 에이전트 세트로 해결하기 어려운 경우에만 예외적으로 추가한다

### 왜 이렇게 결정했는가
- 지금 DIM은 단발성 아이디어보다, 문서·UI·운영·내부 편집 시스템 설계를 반복적으로 정제하는 방식이 더 중요하다
- 같은 에이전트 세트가 프로젝트 맥락을 계속 축적해야 품질과 판단 기준이 흔들리지 않는다

### 기대효과
- 에이전트 검토가 일회성 보고로 끝나지 않고 다음 수정 라운드의 입력으로 누적된다
- 불필요한 에이전트 확장 없이도 DIM 고유의 운영 리듬과 품질 기준을 유지할 수 있다

## 2026-03-20
### 현재 단계
본찰력 정의 고정 단계

### 결정한 내용
- DIM의 모든 해석과 자동 초안 생성은 `본찰력`이라는 고유의 비즈니스 분석 철학에 기반한다고 명시한다
- 본찰력은 `무엇이 화제인가`가 아니라 `무엇이 구조를 바꾸는가`를 판별하는 시선으로 정의한다
- 이후 편집 시스템, AI 초안 생성, 공개 피처 문법 모두 본찰력 규칙을 따라야 한다

### 왜 이렇게 결정했는가
- DIM의 차별점은 단순한 업계 요약이나 인사이트 문장이 아니라, 구조와 의도, 힘의 흐름을 다시 읽는 판단 체계에 있다
- 본찰력을 문서상으로 고정하지 않으면 자동 초안 생성이나 향후 편집 판단이 일반적인 해설형 문장으로 흐를 위험이 크다

### 기대효과
- DIM식 해석의 기준이 브랜드 문장, 편집 문장, 자동 초안 생성 규칙에 일관되게 적용된다
- 에이전트와 편집 시스템이 같은 철학을 기준으로 재귀적으로 개선될 수 있다

### 보류한 것
- 본찰력 규칙을 실제 OpenAI 생성 파이프라인에 연결하는 구현

### 다음 액션
- 본찰력 규칙 기반 draft generator 구현
- 스타일 예시 retrieval과 규칙 엔진 결합

## 누적 핵심 판단

### 편집/철학
- DIM은 뉴스룸이 아니다
- DIM은 프로젝트 제출을 받아 편집과 해석을 거쳐 피처로 정리한다
- 해석 권위가 검색 유입보다 먼저다

### 권위/아카이브
- 홈과 `/articles`는 하나의 매거진 시스템으로 움직여야 한다
- 피처는 단발성 소개가 아니라 축적 자산이어야 한다

### 구현/운영
- MDX 기반 운영을 유지하고 무거운 CMS는 미룬다
- Preview는 Pages, production은 Workers로 분리한다
- 피처는 지식 객체처럼 관리하고, direct answer / evidence / entity clarity를 강화한다
- 내부 편집 시스템은 CMS보다 intake pipeline이 먼저다
- 원본 제출, AI 산출물, 최종 편집본은 분리 보관한다
- 기존 DIM 에이전트들과의 반복 협업과 재귀적 자기개선을 운영 기본값으로 둔다

### 수익/확장
- 권위 형성 전 수익화는 보류한다
- 제출 기반 편집 구조는 장기 확장 입구로 본다
