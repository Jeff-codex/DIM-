# MASTER OPERATING RULES

이 문서는 DIM 전용 에이전트 팀을 운영할 때 쓰는 상위 규칙이다.

## 1. 운영 철학

에이전트는 아이디어를 부풀리는 존재가 아니라,
마스터의 판단과 Codex 실행을 더 빠르고 정확하게 만드는 운영 모듈이다.

모든 출력은 아래 기준을 따라야 한다.

1. 길게 설명하지 말 것
2. 지금 필요한 것과 나중에 미룰 것을 분리할 것
3. Codex가 바로 만들 수 있는 단위로 쪼갤 것
4. DIM의 권위 축적에 직접 연결되는 제안만 남길 것
5. 뉴스룸/CMS/스타트업 랜딩페이지 문법으로 돌아가지 말 것

## 2. DIM 프로젝트 정의

DIM은 뉴스룸이 아니다.
DIM은 제보 사이트도 아니다.
DIM은 산업, 서비스, 런칭, 구조 변화를 해석하는 에디토리얼 인텔리전스 매거진이다.

현재 공개 제품의 기본 흐름은 아래로 고정한다.

- 프로젝트 제출
- 편집 및 해석
- 콘텐츠 발행

## 3. 현재 단계 인식

지금 DIM은 `초기 아이디어 단계`가 아니다.
현재 단계는 `1차 공개 버전 + 매거진 문법 리디자인 이후 정교화 단계`다.

이미 확보된 것:

- 공개 라우트: `/`, `/articles`, `/articles/[slug]`, `/about`, `/submit`
- MDX 기반 피처 운영
- 대표 이미지 체계
- JSON-LD 기반 SEO 바닥
- Pages preview / Workers production 분리
- 9works 레퍼런스를 번역한 매거진형 아카이브 문법

따라서 에이전트는 기본 IA를 다시 발명하기보다,
현재 구조를 더 DIM답게 좁히고 권위를 강화하는 제안을 우선해야 한다.

## 4. 기본 호출 순서

기본 직렬 순서는 아래와 같다.

1. 하린 `GA-1`
2. 유나 `DS-1`
3. 시아 `DO-1`
4. 다인 `RO-1`
5. 세린 `MO-1`

## 5. 호출 규칙

### 규칙 A
한 번에 모든 에이전트를 동시에 쓰지 않는다.
현재 병목과 가장 가까운 2~3개만 먼저 호출한다.

### 규칙 B
세린 `MO-1`은 항상 마지막에 호출한다.
세린은 분석 생산자가 아니라 결론 압축기다.

### 규칙 C
시아 `DO-1` 없이 아이디어만 확장하지 않는다.
Codex가 지금 구현할 수 없는 제안은 우선순위를 낮춘다.

### 규칙 D
DIM의 현재 용어를 유지한다.

- `피처`
- `피처 제안`
- `프로젝트 제출`
- `제출 기반 편집`
- `피처 아카이브`

아래 표현은 기본 금지다.

- `제보`
- `외부 기고`
- `기고 관리`
- `발행 플랫폼`
- `퍼블리싱 시스템`
- `DEPTH` 단독 표기

## 6. 지금 우선순위 프레임

현재 제안은 아래 네 기준으로 본다.

1. Impact — 권위와 제품 인상에 미치는 영향
2. Feasibility — 지금 repo에서 구현 가능한가
3. Clarity — 범위와 완료 기준이 분명한가
4. Leverage — 이후 시리즈, 허브, 운영 루프에 재사용 가능한가

## 7. 주간 운영 루프

### 주초

- `DIM/WEEKLY_BRIEF.md` 갱신
- 가장 큰 병목 1개 지정
- 이번 주 목표 3개만 적기

### 실행 중

- 필요한 에이전트만 호출
- `DIM/BUILD_QUEUE.md` 우선순위 갱신
- Codex로 실제 구현 진행

### 주후반

- 결과를 `DIM/DECISION_LOG.md`에 기록
- 다음 주 병목 다시 정의

## 8. Codex 브리프 작성 규칙

Codex에 넘길 브리프는 반드시 아래를 포함한다.

- 목적
- 구현 범위
- 입력과 출력
- 현재 상태와 바뀌는 상태
- 관련 UI/콘텐츠 구조
- 제외 범위
- 완료 기준

## 9. 지금 버려야 할 제안

- 권위와 무관한 기능 확장
- 초기 단계에서 과도하게 무거운 CMS/D1/R2 설계
- 피처보다 화려한 UI 효과
- 검색량만 보고 주제를 넓히는 제안
- 제출 구조가 아직 비활성인데 운영이 다 된 것처럼 보이게 하는 제안

## 10. 최종 판단 규칙

마스터는 항상 아래 순서로 판단한다.

1. 이게 DIM의 본질과 맞는가
2. 지금 단계에서 필요한가
3. 현재 repo에서 구현 가능한가
4. 권위, 축적, 재방문, 제출 유입에 연결되는가
5. 지금 하지 않는 편이 더 나은가
