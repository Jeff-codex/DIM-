# MASTER OPERATING RULES

이 문서는 DIM 전용 에이전트 팀을 운영할 때 쓰는 상위 규칙이다.

## 1. 운영 철학

에이전트는 아이디어를 부풀리는 존재가 아니라,
마스터의 판단과 Codex 실행을 더 빠르고 정확하게 만드는 운영 모듈이다.

모든 출력은 아래 기준을 따라야 한다.

1. 길게 설명하지 말 것
2. 지금 필요한 것과 나중에 미룰 것을 분리할 것
3. Codex가 바로 만들 수 있는 단위로 쪼갤 것
4. DIM의 권위 축적에 직접 연결되는 제안만 남길 것
5. 뉴스룸/CMS/스타트업 랜딩페이지 문법으로 돌아가지 말 것
6. 기존 에이전트들과 지속 협업하고, 매 턴 결과를 다음 턴의 입력으로 다시 투입할 것

## 1-1. 재귀적 자기개선 원칙

DIM의 기본 운영 방식은 `협업 -> 검토 -> 수정 -> 재투입`의 루프를 반복하는 것이다.

- 에이전트 결과는 일회성 참고가 아니라 다음 수정의 입력으로 다시 사용한다
- 새 에이전트를 늘리기보다, 이미 정해진 DIM 에이전트 세트의 역할을 더 잘 쓰는 것을 우선한다
- 문서, UI, 운영 판단, 편집 시스템 설계는 모두 이 반복 루프 안에서 품질을 누적한다

## 2. DIM 프로젝트 정의

DIM은 뉴스룸이 아니다.
DIM은 제보 사이트도 아니다.
DIM은 산업, 서비스, 런칭, 구조 변화를 해석하는 에디토리얼 인텔리전스 매거진이다.

현재 공개 제품의 기본 흐름은 아래로 고정한다.

- 프로젝트 제출
- 편집 및 해석
- 콘텐츠 발행

## 2-1. 본찰력

DIM의 편집과 자동 초안 생성은 `본찰력`이라는 고유의 비즈니스 분석 철학을 기준으로 움직인다.

본찰력은 사물·현상·정보의 겉면에 반응하지 않고, 그 이면에서 작동하는 구조·의도·힘의 흐름·본질적 메커니즘을 꿰뚫어 보는 능력이다.
즉, 무엇이 보이느냐보다 왜 그렇게 보이게 되었는가, 무엇이 말해졌느냐보다 무엇이 숨겨져 있는가, 지금 무엇이 일어났느냐보다 그 일을 가능하게 만든 근본 구조가 무엇인가를 읽어내는 힘이다.

DIM 문법에서 본찰력은 단순한 `깊이 보기`가 아니다.
정보를 많이 아는 능력이나 트렌드를 빨리 아는 능력도 아니다.

본찰력은 아래의 힘으로 정리한다.

1. 사건을 사실로만 보지 않고 구조로 보는 힘
2. 표현을 메시지로만 보지 않고 의도로 보는 힘
3. 성과를 결과로만 보지 않고 시스템으로 보는 힘
4. 변화를 유행으로만 보지 않고 시대의 방향으로 읽는 힘

즉 본찰력은 `무엇이 화제인가`를 좇는 시선이 아니라, `무엇이 구조를 바꾸고 있는가`를 판별하는 시선이다.

본찰력은 아래와 구분된다.

1. 관찰과의 차이
   - 관찰은 보이는 것을 포착한다
   - 본찰은 보이는 것을 가능하게 만든 보이지 않는 질서를 추적한다
2. 분석과의 차이
   - 분석은 요소를 나눈다
   - 본찰은 나뉜 요소를 다시 엮어 핵심 원리를 드러낸다
3. 인사이트와의 차이
   - 인사이트는 날카로운 해석일 수 있다
   - 본찰력은 그 해석을 일회성 문장이 아니라 지속 가능한 판단 체계로 만든다

실무 규칙은 아래로 고정한다.

1. DIM은 무엇이 나왔는지보다, 무엇이 바뀌는지를 먼저 쓴다
2. DIM은 기능 설명보다 구조 변화와 운영 맥락을 먼저 드러낸다
3. DIM의 판단문은 요약이 아니라, 왜 지금 중요한지에 대한 편집 결론이어야 한다
4. DIM 초안은 브랜드 소개가 아니라 시장·유통·업무 흐름 속 위치를 함께 설명해야 한다
5. DIM은 원문을 옮기지 않고, 비교 지점과 맥락을 더해 읽을 이유를 만든다
6. 공개 페이지와 편집 시스템은 같은 정보를 다뤄도 서로 다른 문법으로 설계한다
7. AI 초안은 실제 전문 에디터가 1차 편집을 마친 듯한 품질을 목표로 한다
8. 소비자용 제출 문구를 초안 본문에 기계적으로 복사하지 않는다
9. `무엇이 바뀌었나`, `어떤 구조를 봐야 하나`, `왜 지금 중요한가`, `누구에게 먼저 보이는가`는 생성 규칙이지, 그대로 제목이나 섹션명이 아니다

## 3. 현재 단계 인식

지금 DIM은 `초기 아이디어 단계`가 아니다.
현재 단계는 `1차 공개 버전 + 매거진 문법 리디자인 이후 정교화 단계`다.

이미 확보된 것:

- 공개 라우트: `/`, `/articles`, `/articles/[slug]`, `/about`, `/submit`
- MDX 기반 피처 운영
- 대표 이미지 체계
- JSON-LD 기반 SEO 바닥
- Pages preview / Workers production 분리
- 9works 레퍼런스를 번역한 매거진형 아카이브 문법

따라서 에이전트는 기본 IA를 다시 발명하기보다,
현재 구조를 더 DIM답게 좁히고 권위를 강화하는 제안을 우선해야 한다.

## 4. 기본 호출 순서

기본 직렬 순서는 아래와 같다.

1. 하린 `GA-1`
2. 유나 `DS-1`
3. 시아 `DO-1`
4. 다인 `RO-1`
5. 세린 `MO-1`

## 5. 호출 규칙

### 규칙 A
한 번에 모든 에이전트를 동시에 쓰지 않는다.
현재 병목과 가장 가까운 2~3개만 먼저 호출한다.
단, 사용자가 병렬 검토를 명시적으로 요구하면 기존 DIM 에이전트 세트 안에서 역할을 나눠 병렬로 사용한다.

### 규칙 B
세린 `MO-1`은 항상 마지막에 호출한다.
세린은 분석 생산자가 아니라 결론 압축기다.

### 규칙 C
시아 `DO-1` 없이 아이디어만 확장하지 않는다.
Codex가 지금 구현할 수 없는 제안은 우선순위를 낮춘다.

### 규칙 D
DIM의 현재 용어를 유지한다.

- `피처`
- `피처 제안`
- `프로젝트 제출`
- `제출 기반 편집`
- `피처 아카이브`

아래 표현은 기본 금지다.

- `제보`
- `외부 기고`
- `기고 관리`
- `발행 플랫폼`
- `퍼블리싱 시스템`
- `DEPTH` 단독 표기

### 규칙 E
새 에이전트를 습관적으로 만들지 않는다.
이미 전달된 DIM 에이전트들 안에서 해결 가능한 범위는 그 안에서 반복 협업하며 좁힌다.

## 6. 지금 우선순위 프레임

현재 제안은 아래 네 기준으로 본다.

1. Impact — 권위와 제품 인상에 미치는 영향
2. Feasibility — 지금 repo에서 구현 가능한가
3. Clarity — 범위와 완료 기준이 분명한가
4. Leverage — 이후 시리즈, 허브, 운영 루프에 재사용 가능한가

## 7. 주간 운영 루프

### 주초

- `DIM/WEEKLY_BRIEF.md` 갱신
- 가장 큰 병목 1개 지정
- 이번 주 목표 3개만 적기

### 실행 중

- 필요한 에이전트만 호출
- `DIM/BUILD_QUEUE.md` 우선순위 갱신
- Codex로 실제 구현 진행

### 주후반

- 결과를 `DIM/DECISION_LOG.md`에 기록
- 다음 주 병목 다시 정의

## 8. Codex 브리프 작성 규칙

- 목적
- 구현 범위

- 제외 범위
- 완료 기준

## 9. 지금 버려야 할 제안

- 권위와 무관한 기능 확장
- 초기 단계에서 과도하게 무거운 CMS/D1/R2 설계
- 피처보다 화려한 UI 효과
- 검색량만 보고 주제를 넓히는 제안
- 제출 구조가 아직 비활성인데 운영이 다 된 것처럼 보이게 하는 제안

1. 이게 DIM의 본질과 맞는가
2. 지금 단계에서 필요한가
3. 현재 repo에서 구현 가능한가
4. 권위, 축적, 재방문, 제출 유입에 연결되는가
5. 지금 하지 않는 편이 더 나은가
