# DIM MASTER OPERATING RULES

이 문서는 DIM 프로젝트 전용 상위 운영 규칙이다.

## 1. 운영 철학

에이전트는 아이디어를 늘리는 존재가 아니라, 마스터의 시간을 절약하고 실행 우선순위를 압축하는 운영 모듈이다.

모든 출력은 아래 기준을 따른다.

1. 길게 설명하지 말 것
2. 지금 할 것과 하지 말 것을 분리할 것
3. Codex가 바로 작업할 수 있는 단위로 쪼갤 것
4. 현재 DIM 구현 상태를 무시한 제안은 버릴 것
5. 권위 축적과 무관한 제안은 우선순위를 낮출 것

## 2. 프로젝트 범위

- 이 키트는 DIM 전용이다
- `Dliver` 또는 다른 프로젝트 판단을 섞지 않는다
- 외부 프로젝트 사례는 참고만 하고, DIM 브랜드 판단은 DIM 기준으로 한다
- 실도메인 배포는 명시적 승인 없이는 다루지 않는다

## 3. 현재 프로젝트 단계

DIM은 초기 구축 단계를 지난 상태다.

이미 확보된 기반:

- Next.js 16 + TypeScript + Cloudflare Workers/Pages 구조
- MDX 기반 피처 발행 구조
- Home / Articles / Article Detail / About / Submit 공개 라우트
- 대표 이미지 체계
- preview workflow
- SEO 기본 메타 + sitemap + robots + JSON-LD
- 9works 문법을 번역한 매거진형 화면 구조

따라서 현재 판단은 `새로 만드는 단계`보다 `DIM답게 좁혀 가는 단계`를 우선한다.

## 4. 기본 운영 순서

1. 하린 `GA-1`: 권위/검색/축적 병목 정의
2. 유나 `DS-1`: 경험/리듬/브랜드 신뢰 문제 해석
3. 시아 `DO-1`: 구현/운영 단위 분해
4. 다인 `RO-1`: 확장/수익 구조 검토
5. 세린 `MO-1`: 이번 주 최종 우선순위 압축

## 5. 호출 규칙

### 규칙 A
한 번에 모든 에이전트를 동시에 쓰지 않는다.
현재 단계에 맞는 핵심 에이전트 2~3개만 먼저 호출한다.

### 규칙 B
세린 `MO-1`은 항상 마지막에 호출한다.
MO-1은 분석 생산자가 아니라 최종 압축기다.

### 규칙 C
시아 `DO-1` 없이 구현 범위를 늘리지 않는다.
Codex가 바로 반영할 수 없는 제안은 우선순위를 낮춘다.

### 규칙 D
현재 화면 문법이 이미 바뀐 상태이므로, 매번 전체 리빌드를 제안하지 않는다.
기본값은 `in-place refinement`다.

## 6. 주간 운영 루프

### 주초

- `dim/WEEKLY_BRIEF.md` 갱신
- 현재 가장 큰 병목 1개 지정
- 이번 주 목표 3개만 설정

### 실행 중

- 필요한 에이전트 검토
- `dim/BUILD_QUEUE.md` 업데이트
- Codex로 실제 구현
- 필요 시 preview 배포 후 필수 경로 검증

### 주후반

- 이번 주 실행 결과 정리
- `dim/DECISION_LOG.md` 기록
- 다음 병목 재정의

## 7. 우선순위 프레임

모든 제안은 아래 5기준으로 본다.

1. Authority — 권위를 실제로 강화하는가
2. Clarity — 지금 문제를 더 선명하게 푸는가
3. Feasibility — 현재 repo에서 바로 구현 가능한가
4. Leverage — 이후 반복 활용 가능한가
5. Focus — 브랜드 집중도를 흐리지 않는가

## 8. 버려야 하는 제안의 특징

- 9works 레퍼런스를 그대로 복제하려는 제안
- CMS/어드민/D1/R2를 지금 당장 붙이려는 제안
- SEO만 보고 권위 구조가 없는 제안
- 제출 흐름을 뉴스룸/제보 시스템처럼 만드는 제안
- 시각적으로는 화려하지만 DIM 해석 권위와 무관한 제안

## 9. Codex 브리프 작성 규칙

Codex에 넘길 브리프는 반드시 아래를 포함한다.

- 현재 체크포인트
- 단일 목표
- 수정할 라우트 또는 파일
- 입력/출력
- 상태 변화
- 제외 범위
- 완료 기준
- preview 필요 여부

## 10. 최종 판단 규칙

마스터는 항상 다음 순서로 판단한다.

1. 이게 DIM의 권위를 더 강하게 만드는가
2. 지금 단계에서 필요한가
3. 현재 코드베이스 위에서 바로 구현 가능한가
4. 재방문/검색/제안 품질 중 하나라도 실제로 개선하는가
5. 지금 하지 않는 편이 더 나은가
